## What Doesn't Work

- **2026-07-11** — `eval_runs.cost_usd` is a real, populated DB column (set on every `insertRun` call in `service.ts`, both success and per-case-error paths) but the entire eval analytics READ path hardcoded `cost_usd: null` everywhere — `EvalRunBatch` had no `cost_usd` field at all, and `EvalDashboard.current.cost_usd` / `EvalTrendPoint.cost_usd` (which DID have the field in the Zod contract) were still stub-nulled in `analytics.ts`. The gap was purely in the read/aggregation layer, not persistence — `batchRunsWithExpectedForOwner` (`repository.ts`) simply never selected the column. Fix pattern (mirrors the live-run convention in `service.runBatch`: `batchCostUsd += output.costUsd ?? 0`, null contributes 0, batch total is always a number not nullable): add `costUsd: t.evalRuns.costUsd` to the repository select, sum it in `aggregateRunRows`, add non-nullable `cost_usd: z.number()` to `EvalRunBatch` + `EvalCompare.delta` in BOTH `server/src/vendor/shared` and `client/src/vendor/shared` copies (they are NOT auto-synced — confirmed by diffing them, they'd already drifted on unrelated fields), then wire `current.cost_usd`/`trend[].cost_usd` in `_agentDashboard` to the now-real batch values instead of the `null` stub. Evidence: `server/src/modules/eval/analytics.ts`, `server/src/modules/eval/repository.ts:200-221`, `server/src/modules/eval/service.ts:337` (the `?? 0` convention to replicate).

- **2026-07-07 (onboarding)** — Hard-coding empty sections in an LLM-failure fallback throws away completed work: onboarding's `narrativeUnavailable` skeleton set `firstTasks: undefined` even though gap detection had already run deterministically BEFORE the LLM call — genuinely detected First Tasks vanished exactly when the deterministic fallback mattered most. Pattern: an LLM-failure skeleton must carry over every section computed pre-LLM; only the degraded-index path (no facts available) may omit them. Evidence: `server/src/modules/onboarding/service.ts` (`_buildSkeleton`), `analyzers/skeleton.ts`; fixed in commit `469e93f`.

- **2026-07-07** — Resolving "the workspace's repo" with `SELECT … FROM repos WHERE workspace_id = ? LIMIT 1` and no `ORDER BY` is heap-order nondeterministic — live verification got a seed repo whose clone didn't exist on disk, failing attach-time path validation. Pattern now: accept an explicit `repoId` (threaded from the client's `useActiveRepo`), else fall back to `ORDER BY createdAt` preferring a repo whose clone dir `stat()`s. Evidence: `server/src/modules/agents/service.ts` (`resolveCloneRoot`).
- **2026-07-11** — `db/seed.ts` inserted the 4 built-in agents via raw `db.insert(t.agents).values(a)`, bypassing `AgentsRepository.insert()` — the only path that also writes the matching `agent_versions` snapshot row via the (then-private) `snapshotVersion()`. Result: `EvalAnalytics.compare()`'s `agentsRepo.getVersion(agentId, 1)` always returned `undefined` for these agents, so `prompt_diff.old`/`new` were silently `null` and the client's Compare-runs modal showed "No system prompt available" even with real prompts on both eval runs. Any repository method that has a "write row + also snapshot/derive a second row" contract (`insert()` + `snapshotVersion()` here) must be the ONLY insertion path for that entity — a bypass elsewhere (seed scripts, migrations, admin scripts) silently violates the invariant with no compile-time signal, since the second table is optional/nullable by schema. Backfilling later is non-trivial: the missing version-1 snapshot cannot be reconstructed from the entity's CURRENT row if it's since been edited (version bumped, current row now holds the edited config) — must reconstruct from the original creation-time values (here, the seed prompt constants), not from `existing`. Evidence: `server/src/db/seed.ts` (built-in agent loop + Test Quality Reviewer block), `server/src/modules/agents/repository.ts:85-106,148-167`, `server/src/modules/eval/analytics.ts:297-309`.
- **2026-07-13** — `scoring.ts` had 100% of its 29 pre-existing tests green and full line coverage, yet a first Stryker mutation run (72 mutants) found `aggregate()`'s `traces_passed: cases.filter((c) => c.score.pass).length` survived when mutated to `cases.length` — every existing `aggregate()` call in the suite happened to build a batch where ALL cases pass, so the buggy mutant (which counts every case as "passed") produced an output identical to the correct code. Green tests + full coverage do NOT prove a test would catch this class of regression; only a MIXED pass/fail batch in one `aggregate()` call distinguishes the two. Fixed by adding exactly that test — mutation score 94.44%→95.83% (68→69/72 killed). 3 mutants remain survived (an off-by-one on `regionsIntersect`'s inclusive boundary check, and two `some`↔`every` swaps on the noise-matching predicates in `scoreCase`) — not yet addressed, left for a future pass. Evidence: `server/src/modules/eval/scoring.ts:163`, `server/src/modules/eval/scoring.test.ts` ("traces_passed counts only passing cases, not every case in the batch"), regenerate the full report via `pnpm test:mutation` (`server/reports/mutation/`, gitignored).

- **2026-07-13** — A skill eval case can pass "by luck" and then fail forever after, with the failure having NOTHING to do with the skill config: the `missing-owner-guard` eval case (`pr-quality-rubric` skill) stored an `input_diff` whose hunk header claimed `@@ -119,10 +119,10 @@` but every body line was unchanged context — zero `+`/`-` lines, a no-op diff that never actually showed the guard being removed. A diff-scoped reviewer has nothing legitimate to flag when nothing changed, so whether the model happened to comment on the unchanged context was pure chance; worse, the case's expected region (129-132) sat OUTSIDE the hunk's declared new-line range (119-128), so `reviewer-core/src/grounding.ts`'s gate would silently drop a correctly-placed finding too. Diagnosis method that separated "config bug" from "fixture bug": queried `eval_runs` for the case directly (ran_at history showed pass at v1, fail at v2, still fail at v3) AND diffed `skill_versions.body` for v1 vs v3 (`v1 === v3` came back `true`, a byte-identical string compare) — proving the user's config revert was correct and the flakiness was upstream in the fixture, not downstream in a broken revert. Fixed by rewriting `input_diff` to a real unified diff (`-` prefixed deletion of the 15-line guard block, rest as context) and confirming with `parseUnifiedDiff` that the hunk's `newLineNumbers` now covers the expected region. Before trusting an eval case's pass/fail history, verify with `parseUnifiedDiff` that its `expected_output` region actually falls inside the diff's real hunk line coverage — a case can look like flaky config drift when it's actually an ungrounded fixture. Evidence: `server/src/modules/eval/repository.ts` (`eval_cases.input_diff` for case id `3d9778e2-2c44-495f-8190-e54c865a22e2`), `reviewer-core/src/grounding.ts` (`buildLineIndex`, `rangeIntersects`), `server/src/adapters/git/diff-parser.ts` (`parseUnifiedDiff`).

- **2026-07-13** — The eval-cases module shipped with `POST /eval-cases` (create), `DELETE /eval-cases/:id`, and `POST /eval-cases/:id/run`, but no update route — `EvalRepository` had `insertCase`/`deleteCase`/`getCase` and no `updateCase`. `EvalCaseModal`'s Save button is shared between "new case" and "edit case" (gated by whether `caseId` is passed), but with no update endpoint to call, Save in edit mode had no choice but to call `createEvalCase` unconditionally — silently duplicating the case on every edit instead of updating it. The modal's own header comment documented this as an intentional invariant ("always calls `createEvalCase` on Save regardless of origin"), which masked the gap from casual reading — worth checking, when a component branches UI on an "edit mode" flag, whether the corresponding write endpoint for that mode actually exists. Fixed by adding `EvalRepository.updateCase` + `service.updateCase` + `PUT /eval-cases/:id` + client `updateEvalCase`, with `EvalCaseModal.handleSave` branching on `caseId`. Evidence: `server/src/modules/eval/repository.ts` (`updateCase`), `server/src/modules/eval/service.ts` (`updateCase`), `server/src/modules/eval/routes.ts` (`PUT /eval-cases/:id`), `client/src/components/EvalCaseModal/EvalCaseModal.tsx` (`handleSave`).

## Recurring Errors & Fixes

- **2026-07-07** — A `*/` sequence inside a JSDoc comment (even wrapped in backticks, e.g. documenting a glob or `test file suffix`) terminates the `/* … */` block comment early — the rest of the file then parses as garbage (here: an unterminated template literal) with tsc errors pointing far from the real cause. Fix: never write `*/` inside a block comment; spell it out (e.g. `(* slash)`). Evidence: `server/src/modules/onboarding/service.ts` (`checkDocCoverage` docblock).

- **2026-06-14** — Adding a required field to a Zod contract (`RunStats.cost_usd`) breaks the inline fixture in `server/test/contracts.test.ts` (RunTrace parse). Update the `stats: {…}` fixture in the same change. Evidence: `server/test/contracts.test.ts:160`.
- **2026-07-07** — `db/migrations/meta/` is missing snapshots 0012–0014, so `db:generate` diffs the schema against snapshot 0011 and re-detects historical column renames — an interactive TTY prompt plus already-applied ALTERs leak into the new migration. Workaround used for 0015: answer prompts via an `expect` PTY, then strip everything but the genuinely new statements from the generated SQL. `0015_snapshot.json` now captures the full 42-table schema, so future generates diff cleanly. Evidence: `server/src/db/migrations/meta/_journal.json`, `0015_careless_famine.sql`.
- **2026-07-07** — A running dev server does NOT auto-apply new migrations: after a schema-adding commit, routes touching the new tables 500 with `relation "…" does not exist` until a manual `cd server && npm run db:migrate`. If a brand-new endpoint 500s with `internal_error`, check for unapplied migrations first. Evidence: live verification of `agent_documents`/`skill_documents` (migration 0015).
- **2026-06-25** — `reviews/repository/pull.repo.ts` contains hidden `upsertIntent`/`getIntent` helpers that mirror the `Intent` contract shape. When `Intent` fields are renamed (e.g. `intent` → `summary`), this file must be updated alongside the contract — it's easy to miss because it lives inside the `reviews` module, not in `modules/intent/`. Evidence: `server/src/modules/reviews/repository/pull.repo.ts:49-68`.
- **2026-07-16** — Merging two lesson branches that each generated their own migrations can desync `drizzle.__drizzle_migrations` even though the SQL content is fine: `db:migrate` failed with `column "multi_agent_run_id" of relation "agent_runs" already exists` on a freshly-merged branch (`lesson/07-integration` ← `lesson/07-multi-agents-review` + `lesson/07-export-to-ci`). Cause — drizzle-orm's postgres-js migrator only compares the SINGLE max `created_at` in the tracking table against each migration file's journal `when` timestamp; it never checks per-file hashes to decide what's already applied. The merge renumbered the multi-agent-run migration to `0022` with a NEW journal timestamp that postdates `0019`–`0021`, but its DDL had already run earlier (on the source branch) and was recorded under an OLDER `created_at` that now sits chronologically *before* `0019`–`0021`'s rows — so the migrator's "newest timestamp" check thought `0022` had never run. Diagnosis: `shasum -a 256` on the failing `.sql` file matched a `hash` already present in `drizzle.__drizzle_migrations` (`docker exec devdigest-postgres psql -U devdigest -d devdigest -c "select id,hash,created_at from drizzle.__drizzle_migrations order by created_at"`), and `\d agent_runs` / `\d multi_agent_runs` confirmed every statement in the file was already live. Fix: no schema change — `INSERT INTO drizzle.__drizzle_migrations (hash, created_at) VALUES ('<sha256 of the .sql file>', <journal entry's "when" value>)` so the max-timestamp check passes; this is a bookkeeping-row fix, not a migration edit. General rule: before assuming schema drift on an "already exists" `db:migrate` error after a branch merge, hash-compare the failing migration file against existing `__drizzle_migrations.hash` rows first. Evidence: `server/src/db/migrations/0022_fair_princess_powerful.sql`, `server/src/db/migrations/meta/_journal.json` (entries `idx: 19-22`).
