## What Works

- **2026-07-07** — "Resource not generated yet" tri-state: catch the 404 in the fetcher (`ApiError.status === 404` → return `null`) so TanStack Query lands in the success state with `data === null` — the page then branches cleanly: `null` → first-visit CTA, data → content, `isError` → genuine failure only. No error-object probing in components. Evidence: `client/src/lib/api.ts` (`fetchOnboarding`), `client/src/lib/hooks/onboarding.ts`.

- **2026-06-14** — `formatCost` (`src/lib/cost.ts`) distinguishes MISSING data (`null`/`undefined` → "—") from a genuine zero (`0` → "$0.00"), widens precision for sub-cent values (~2 sig figs), and trims trailing zeros to a 2dp floor ("$0.06" not "$0.060", "$0.0013" not "$0.00"). Reuse it for any per-run money display.
- **2026-07-13** — Testing a Recharts `<Tooltip content={...}>` by simulating real mouse hover in jsdom is unreliable: `ResponsiveContainer` reports 0×0 dimensions under jsdom (visible as a console warning on every chart render in this test suite already), so Recharts' internal active-index/coordinate math has no real geometry to key off, making `fireEvent.mouseOver`/`mouseMove` on the SVG a flaky way to trigger the tooltip. Pattern that works: export the tooltip's content-renderer as its own named component (e.g. `ChartTooltip({active, payload})`) and unit-test IT directly via RTL `render()` with hand-built `{active, payload}` props — bypasses the mouse-simulation problem entirely, deterministic, and still exercises the real formatting logic. Evidence: `client/src/vendor/ui/charts/LineChart.tsx` (`ChartTooltip`), `client/src/vendor/ui/charts/LineChart.test.tsx`.

## What Doesn't Work

- **2026-07-11** — `@devdigest/ui`'s `charts/Sparkline.tsx` (and `MetricCard`, which renders one internally whenever a `trend` prop is passed) divides by `data.length - 1` to place points — a trend array of length 1 produces `0/0 = NaN`, which React then throws into `<path d="MNaN,18.0">` / `<circle cx="NaN">` at render, spamming console errors (caught live via Playwright screenshot verification, not by the component test — jsdom doesn't surface SVG attribute-parse errors as failures). Any caller passing a real (not >=2-length-guaranteed) trend array MUST gate it: `trend={points.length >= 2 ? points : undefined}`. `AgentEvalCard`'s own local Sparkline already guarded this way; `MetricCard` usages must do the same manually since the vendor component doesn't guard internally. Evidence: `client/src/vendor/ui/charts/Sparkline.tsx`, `client/src/app/eval/[agentId]/_components/AgentEvalDetailView.tsx`.
- **2026-07-11** — A client-side "date range" filter (e.g. last 7/30/90 days) that defaults to a narrow window computed against real `Date.now()` can silently render an empty state on first load whenever the underlying data (seed/demo data, or just an agent that hasn't run recently) is older than the window — there's no visible cue that a filter, not an empty dataset, is hiding everything. Default such filters to "All time" and let the user narrow deliberately. Evidence: `client/src/app/eval/[agentId]/_components/AgentEvalDetailView.tsx` (caught before shipping — a 30-day default against 2026-07-11 "now" hid May-2026 seed batches entirely).

- **2026-07-11** — `fetchEvalDashboard()` called with no `agentId` (the null-owner/workspace variant) returns `recent_runs` as bare `EvalRunRecord[]` — per-*case* rows with no agent name/id attached. It cannot power a workspace-wide table that needs to show which agent a row belongs to (e.g. an "all agents" recent-runs view). Fix: drop the null-owner call and instead fan `fetchEvalBatches(agentId)` out across every agent via `useQueries`, tagging each returned `EvalRunBatch` with the agent's own name/id client-side, then merge+sort by `ran_at`. `fetchEvalBatches` is already agent-scoped and already used by `EvalsTab`, so this reuses a proven endpoint instead of extending the dashboard contract. Evidence: `client/src/app/eval/_components/EvalDashboardView.tsx`, `client/src/lib/api.ts:195` (`fetchEvalBatches`).
- **2026-07-07 (later)** — An inline preview panel appended AFTER a long list inside a narrow column (`maxWidth:580`) opens below the fold and reads as "the button does nothing" — the Context tabs' original `<pre>` preview was state-wired correctly yet reported broken for exactly this reason. For any on-demand content preview use the vendor kit `Drawer` (`@devdigest/ui`, `position:fixed; inset:0` right-side overlay — RunTraceDrawer precedent): visibility becomes layout-independent. Also supersedes the `SafeMarkdown` evidence path above: the component was promoted to `client/src/components/SafeMarkdown/` (old `_components/` copy deleted). Evidence: `client/src/app/agents/[id]/_components/AgentEditor/_components/ContextTab/ContextTab.tsx`.
- **2026-07-07** — `vendor/ui/primitives/Markdown` is NOT safe for untrusted (repo-controlled) content: it renders `<a href={href}>` without filtering `javascript:`/`data:` URLs. For untrusted markdown use the `SafeMarkdown` wrapper pattern (custom `a` component with an `isSafeHref` check, no `rehype-raw` so inline HTML stays inert). Evidence: `client/src/vendor/ui/primitives/Markdown.tsx:53-57`, `client/src/app/project-context/_components/SafeMarkdown.tsx`.
- **2026-07-06** — The reference-build `BlastGraph` (Blast tab Graph toggle) graphed ONLY `blast.downstream[0]` and rendered the "nothing to graph" empty state whenever that first entry had 0 callers — even when later symbols had callers/endpoints (PR #7: `downstream[0]` = CodeLine with 0/0, while executeRuns/ReviewRunExecutor each had a caller). Fixed by deriving per-symbol nodes/edges over ALL downstream entries (same attribution as `BlastGraphLightbox`), skipping symbols with nothing downstream; empty state now requires zero edges total. Evidence: `client/src/app/repos/[repoId]/pulls/[number]/_components/BlastRadius/BlastRadius.tsx:96`.
- **2026-06-30** — `reviews.ts` does NOT re-export `usePrIntent` or `useSmartDiff` by default — those hooks live in `intent.ts` and `smartDiff.ts` respectively. `page.tsx` imports both from `reviews.ts` and silently fails at runtime if not bridged. Fix: add `export { usePrIntent, useClassifyIntent } from './intent.js'` and `export { useSmartDiff } from './smartDiff.js'` to `reviews.ts`. Evidence: `client/src/lib/hooks/reviews.ts`.
- **2026-06-30** — Two SmartDiffViewer implementations exist simultaneously: the reference version at `_components/SmartDiffViewer/` (props: `smartDiff, files, commenting?`) and the enhanced version at `src/components/SmartDiffViewer/` (props: `smartDiff, allFindings, files?, onNavigateToFinding`). `DiffTab` must import from `@/components/SmartDiffViewer` (the enhanced one); using the relative `../SmartDiffViewer` path would silently miss the enhanced API or fail if `_components/SmartDiffViewer/` is absent. Evidence: `_components/DiffTab/DiffTab.tsx`.
- **2026-06-30** — `PrBrief.intent` schema diverged from the reference build: current codebase uses `summary: string`; the reference uses `intent: string`. Copying `PrBriefCard` from the reference causes a TS error at `intent.intent` — replace with `intent.summary`. The mismatch also breaks test fixtures. Evidence: `_components/PrBriefCard/PrBriefCard.tsx:42`, `client/src/vendor/shared/contracts/brief.ts:10`.
- **2026-06-17** — The PR-list `tableCard` has `overflow: "hidden"` (`pulls/styles.ts`) which CLIPS absolutely-positioned hover popovers (`FindingsHoverCard`) opening downward from the bottom rows; upper rows render fine (matching the design). `FindingsHoverCard` is dependency-free (anchor wrapper + `position:absolute` panel) — to fully escape the card it would need a portal + `position:fixed` from the anchor's `getBoundingClientRect`. Deferred; not needed for the common case. Evidence: `client/src/components/FindingsHoverCard/`, `pulls/styles.ts:97`.

- **2026-07-11** — `EvalRunBatch`/`EvalDashboard.current.traces_passed`/`traces_total` count per-*case* runs, NOT per-region/finding — a batch of 3 cases has `traces_total: 3` regardless of how many findings each case's `expected_output.regions[]` lists. `runEvalCase(caseId)` (the modal's "Run case" / a row's run icon) generates its OWN isolated `batchId` (see `service.ts` `runCaseOnce`), so it becomes its own 1-trace batch — and since the dashboard's `current`/`delta` are "latest batch vs. previous batch", running ONE case standalone temporarily makes the EvalsTab metrics strip reflect only that single case's recall/precision (e.g. 100%) instead of the aggregate across all cases, until the next full "Run all evals" batch. Confirmed live via Playwright screenshot. Evidence: `server/src/modules/eval/service.ts` (`runCaseOnce`), `client/src/app/agents/[id]/_components/AgentEditor/_components/EvalsTab/EvalsTab.tsx`.
- **2026-07-11** — `EvalDashboard.delta.*` defaults to `0` server-side when there is no prior batch to compare against (`analytics.ts`: `current && prev ? current.x - prev.x : 0`) — a UI that renders any non-null delta as a flat/unchanged badge misreads "no baseline yet" as "genuinely unchanged". Gate delta rendering on `dashboard.trend.length >= 2` (a real prior point exists), not just `dashboard != null`. Evidence: `client/src/app/agents/[id]/_components/AgentEditor/_components/EvalsTab/EvalsTab.tsx`.
- **2026-07-11** — `diff-viewer/helpers.ts`'s `parsePatch()` checks only `raw.startsWith("-")`/`"+"`, so it misclassifies unified-diff file-header lines (`--- a/file`, `+++ b/file`) as `del`/`add` content lines — it's built for hunk bodies only, not a full diff blob with headers. A read-only preview of a raw `input_diff` string (headers + hunks) needs its own line classifier that checks `--- `/`+++ ` prefixes before the `+`/`-` checks. Evidence: `client/src/components/diff-viewer/helpers.ts`, `client/src/components/EvalCaseModal/EvalCaseModal.tsx` (`parseDiffLines`).
- **2026-07-13** — `EvalCaseModal`'s region-row red/invalid styling (`isRowInvalid`) reused the SAME validity check for both "disable Save" and "render error border" — since a fresh blank region row (`file: ""`) is invalid by definition, the New Eval Case modal opened with its lone default row already outlined in red, before any user input. Fix: give `RegionRow` a stable `id` (`crypto.randomUUID()`, not array index) and track a `touchedRegionIds` Set populated inside `updateRegion`; gate the red styling on `isRowInvalid(row) && touchedRegionIds.has(row.id)` while Save-disabling still uses the raw, untouched-agnostic `isRowInvalid`. General pattern: a "some field is required" validity check that also drives visual error styling needs a separate touched/dirty gate, or every fresh/blank row renders as broken on load. Evidence: `client/src/components/EvalCaseModal/EvalCaseModal.tsx` (`isRowInvalid`, `touchedRegionIds`).

## Recurring Errors & Fixes

- **2026-07-11** — A component test that does `vi.mock("@devdigest/ui", () => ({ Skeleton: ... }))` (a full factory replacement, not a partial mock) silently breaks with "Element type is invalid" the moment the component-under-test starts importing any OTHER named export from that barrel (`Button`, `SectionLabel`, `ProgressBar`, `Badge`, `Icon`, …) — those come back `undefined` since the factory only lists `Skeleton`. Fix: every time a component under test picks up a new `@devdigest/ui` import, add a matching stub to that test file's mock factory (plain `<button>`/`<div>`/`<span>` stand-ins are fine; `Icon` needs a `Proxy` since it's accessed as `Icon.<Name>` for arbitrary names). Evidence: `client/src/app/eval/_components/EvalDashboardView.test.tsx`.
- **2026-07-05** — Relative import depth from `pulls/[number]/_components/<Tab>/` to `src/lib/` is SEVEN levels up (`../../../../../../../lib/hooks/brief`), not five — plan docs and intuition routinely undercount because of the `repos/[repoId]/pulls/[number]` nesting. Fix: grep a neighboring component's imports (e.g. `ComposeReviewDrawer`) and copy the depth instead of counting. Evidence: `client/src/app/repos/[repoId]/pulls/[number]/_components/BlastTab/BlastTab.tsx`.

- **2026-06-30** — React warns when `borderColor` (shorthand) and `borderLeftColor` (longhand) appear in the same style object — both affect the same CSS property and React detects the conflict on re-render. Fix: replace `borderColor` with three explicit longhands (`borderTopColor`, `borderRightColor`, `borderBottomColor`) and keep `borderLeftColor` unchanged. Evidence: `client/src/app/repos/[repoId]/pulls/[number]/_components/FindingCard/styles.ts:12-15`.
